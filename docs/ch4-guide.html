# 2. 检查词表覆盖率（UNK 率）
def unk_rate(text, tokenizer):
    ids = tokenizer.encode(text, add_special_tokens=False)
    unk_id = tokenizer.unk_token_id
    return ids.count(unk_id) / len(ids) if ids else 0

# 3. 检查 chat template
msgs = [{"role":"user","content":"你好"}]
prompt = tok.apply_chat_template(msgs,
    tokenize=False, add_generation_prompt=True)

print(f"EOS: {tok.eos_token} → {tok.eos_token_id}")
print(f"PAD: {tok.pad_token} → {tok.pad_token_id}")

# 5. 测量 tokenization 效率（tokens/char）
def compression_ratio(text, tokenizer):
    n_tokens = len(tokenizer.encode(text))
    n_chars = len(text)
    return n_chars / n_tokens  # 越高越好

0.1B 模型配置方案

0.1B（1亿参数）是学习大模型的黄金尺寸：大到足以展示智能涌现，小到可以用一张消费级 GPU 在几小时内完成训练。这里给出三种配置方案，并深入解析每个参数背后的数学原理。

配置hidden_size层数参数量显存（fp16, bs=16）适用场景
Nano2568≈13M≈2GB快速实验，MacBook M 系列
Small5128≈27M≈4GBRTX 3060/4060，快速迭代
Base (0.1B)7688≈100M≈8GBRTX 3080/4080，正式训练
精确参数量计算（hidden=768）：
Embedding 层：768 × 6400 = 4,915,200 ≈ 4.9M（如 tie_word_embeddings=True 则与输出层共享）
每层 Attention（GQA，4个KV头，8个Q头）：
  · W_Q: 768 × (8×96) = 768² = 589,824
  · W_K, W_V: 768 × (4×96) = 294,912 各一份
  · W_O: 768² = 589,824
  · 合计：≈ 1.77M per layer
每层 FFN（SwiGLU，intermediate ≈ 8/3 × 768 ≈ 2048）：
  · gate_proj + up_proj + down_proj: 768×2048×3 ≈ 4.7M per layer
8 层总计：(1.77 + 4.7) × 8 ≈ 51.8M，加 Norm（≈0.02M）和 Embedding ≈ 56-65M
加上未共享的 lm_head（768×6400=4.9M）总参数接近 100M

hidden_size 为什么必须是 2 的幂次？

GPU Tensor Core 的矩阵乘法对尺寸有强烈偏好：

· FP32：矩阵维度是 8 的倍数时最优
· FP16/BF16：维度是 16 的倍数时效率最高
· INT8（量化推理）：需要 32 的倍数

256, 512, 768, 1024 都满足 16 的倍数要求。若用 750 这样的"奇数"尺寸，A100 的矩阵乘单元会有 2.4% 的利用率损失，积累下来影响显著。

768 虽然不是 2^n，但 768 = 3×256，仍是 256 的倍数，在大多数硬件上表现良好。

内存占用分解（0.1B 模型，训练时）

模型权重（FP16）：
100M × 2 bytes = 200MB

梯度（与权重同精度）：
100M × 2 bytes = 200MB

AdamW 优化器状态：
一阶矩（momentum）× 2 + 二阶矩（adaptive LR）× 2 = 400MB
注：Adam 状态通常用 FP32 保存

Activations（batch=16, seq=512）：
约 16 × 512 × 768 × 8层 × 4 bytes ≈ 200MB

总计约 1GB，加上框架开销约 2-3GB

为什么 8 层是 0.1B 的甜点？

Transformer 深度 vs 宽度的研究表明：

· <4 层：无法学习多步推理，对话质量差
· 6-12 层：对于 <1B 参数的模型，是实验验证的"甜点区"
· >16 层（0.1B规模）：梯度回传路径过长，训练不稳定，需要 warm-up 和更谨慎的初始化

GPT-2 Small（117M）选择了 12 层×768，DistilGPT-2（82M）用 6 层×768，MiniMind 的 8 层×768 是两者之间的合理选择。

Chinchilla 计算最优数据量

Hoffmann et al. 2022（Chinchilla 论文）给出经验公式：

最优 tokens ≈ 20 × N_params

· 13M Nano 模型：2.6亿 tokens（约 500MB 文本）
· 27M Small 模型：5.4亿 tokens（约 1GB 文本）
· 100M Base 模型：20亿 tokens（约 4GB 文本）

低于此数量训练是"欠拟合"；远超此数量也有收益，但边际效用递减。许多开源小模型用 300B+ tokens 训练 0.1B 模型——这叫"overtrained but better for inference"。

同类模型参数量对比

GPT-2 Small（OpenAI, 2019）
117M 参数，12层，768 hidden，12头，无 GQA

DistilGPT-2（HuggingFace, 2019）
82M 参数，6层，768 hidden，知识蒸馏自 GPT-2

OPT-125M（Meta, 2022）
125M 参数，12层，768 hidden，ReLU FFN

MiniMind Base（本项目）
~100M 参数，8层，768 hidden，SwiGLU + GQA + RoPE
架构比以上三者都更现代（借鉴 LLaMA-2 设计）

宽度 vs 深度的工程权衡

宽度↑（增大 hidden_size）：
· 每层看到更多特征表示
· 并行计算效率更高（矩阵乘法利用率高）
· Attention 的 O(n²d) 中 d 增大，显存线性增长

深度↑（增加层数）：
· 更深的多步推理能力
· 梯度传播路径更长，需要 skip connection
· 推理延迟增加（层间串行）

实践建议：先增宽度，再增深度。0.1B → 0.3B 时，优先把 hidden 从 768 增到 1024，而不是把层数从 8 加到 16。

CHAPTER 4 · 实战指南

三阶段训练路径

现代大模型训练分三个阶段，每个阶段的目标、数据、超参完全不同。跳过任一阶段都有具体的失败模式。理解"为什么"比记住"怎么做"更重要。

阶段 1：预训练（Pre-training）— 学会语言 目标：最大化 P(x_t | x_1, ..., x_{t-1})，等价于最小化交叉熵损失 L = -1/T · Σ_t log P_θ(x_t | x_{评估：perplexity 下降；随机续写文本语法连贯 阶段 2：指令微调（SFT — Supervised Fine-tuning）— 学会对话 目标：让模型学会"根据 prompt 生成 completion" 只对 completion 部分计算 loss（prompt 部分 label=-100 mask 掉） 数据：高质量对话 QA 对（需要标注或自动生成） 数据量：10万-100万条对话（质量远比数量重要） 学习率：从预训练 lr 的 1/3 开始（2e-4），防止破坏已学知识 批大小：16-64（较小，因为数据更珍贵） 训练时间：2-12 小时 评估：问答准确率；指令遵循能力；MT-Bench 评分 阶段 3（可选）：对齐（RLHF/DPO）— 学会有用且无害 目标：让模型输出符合人类偏好（有帮助、无害、真实） DPO 损失函数： L_DPO = -E[log σ(β·log(π_θ(y_w|x)/π_ref(y_w|x)) - β·log(π_θ(y_l|x)/π_ref(y_l|x)))] y_w = chosen（好的回复），y_l = rejected（差的回复） β 控制偏离参考模型的程度（通常 0.1-0.5） π_ref = SFT 模型（冻结作为基准） 数据量：1万-10万条偏好对 训练时间：1-4 小时 评估：WinRate（与 SFT 对比的胜率）；人工盲评
跳过预训练的失败模式：直接从随机初始化做 SFT，就像让一个完全不识字的人去参加辩论课。SFT 数据里的每个词对模型都是陌生的，模型无法"借力"已有的语言知识。实验表明，跳过预训练需要 50-100 倍更多 SFT 数据才能达到同等效果，而且往往仍然无法匹敌。
跳过 SFT 的失败模式：纯预训练的模型会"续写"而不是"回答"。你问"巴黎的首都是哪里？"，它可能回复"…，因此法国的文化中心一直是…"。它学会了语言规律，但不知道该在什么时候停止、如何回答具体问题。
涌现能力（Emergent Abilities）出现的参数量阈值（经验值）：
· 基本语法和词义理解：~10M 参数
· 简单问答（Who/What/When）：~100M 参数
· 多步推理（数学应用题）：~1B 参数开始涌现，7B+ 稳定
· 代码生成（非平凡程序）：~1B 参数
· 指令遵循（无需示例）：~1B 参数
· 思维链推理（CoT）：~100B 参数时最明显
0.1B 模型能做：流畅中文对话、简单问答、格式化输出。不要期待复杂推理。

预训练数据混合比例的依据

60% 网页文本：覆盖最广泛的话题和写作风格，但质量参差不齐。CommonCrawl + C4 + OpenWebText

20% 代码：即使不做代码任务，代码数据也能提升逻辑推理和结构化输出能力（Google Minerva 研究证明了这点）

10% 书籍：长文本、连贯叙事、高密度知识。PG-19、Books3

10% 学术/百科：精确事实知识、定义性写作。arXiv、Wikipedia

RLHF 方法对比：何时用哪种？

PPO（Proximal Policy Optimization）
优点：理论上最优；缺点：需要额外训练奖励模型，实现复杂，内存需求高（4个模型同时在GPU）。适合：大规模生产环境

DPO（Direct Preference Optimization）
优点：无需奖励模型，实现简单，稳定。缺点：需要成对偏好数据。适合：学术研究、小规模项目

ORPO（Odds Ratio Preference Optimization）
优点：无需参考模型，SFT 和对齐一步完成。适合：计算资源有限

SimPO（Simple Preference Optimization）
优点：比 DPO 更简单，不需要参考模型。2024 年新方法，表现接近 PPO

2 周入门计划（从零到有模型）

第1-2天 环境配置，运行官方推理脚本，理解代码结构
第3-4天 准备 1 万条 SFT 数据（用 GPT-4 或公开数据集），理解 Parquet 格式
第5天 用 MiniMind 预训练权重做 SFT，在本地小模型上跑通
第6-7天 租 GPU（A100 1小时=$2.5），跑完整 SFT，观察 loss 曲线
第8-9天 评估模型输出质量，找出缺陷，增加针对性数据
第10-12天 迭代一次 SFT，尝试调整学习率和 epoch 数
第13-14天 尝试 DPO：收集 500 条偏好对，做一次对齐训练

DPO 损失函数详解

L_DPO 的直觉理解：

· π_θ(y_w|x)/π_ref(y_w|x)：当前模型相对参考模型，对"好回复"的概率比。希望这个比值 >1（当前模型更倾向于好回复）
· π_θ(y_l|x)/π_ref(y_l|x)：当前模型对"差回复"的概率比。希望这个比值 <1
· β 是温度参数：β 越大，越强迫模型偏离参考；β 越小，越保守。通常取 0.1-0.5
· σ 是 sigmoid 函数，把差值映射到 0-1 之间，类似二分类损失

自己爬取数据

用 BeautifulSoup 爬取网页时的合规原则：

1. 检查 robots.txt（如 zhihu.com/robots.txt）
2. 每次请求间隔 1-3 秒（rate limiting）
3. 设置 User-Agent 标识爬虫身份
4. 只爬公开内容，不绕过登录墙

去重：用 URL 和内容的 MD5 哈希建立 set，O(1) 判重。100万 URL 的哈希表约 50MB 内存。

用 LLM 生成合成数据

完整工作流：
1 准备 20-50 条种子问题（覆盖你的领域）
2 提示词："以下是 {n} 个关于 {domain} 的多样化问题：{seeds}。请生成 20 个类似但不重复的新问题，涵盖不同难度和角度"
3 对每个问题生成详细回答（指定格式、长度、风格）
4 质量过滤：用小模型打分，或用规则（长度/关键词/格式）自动过滤

成本参考：GPT-4o 生成 1000 条对话约 $10-15；GPT-3.5-turbo 约 $1-2

领域专用数据源

医疗：PubMed（3300万摘要，免费）；风险：需要领域专家审查，不可直接做医疗建议

法律：CourtListener/Case Law（800万判决书）；价值极高但格式复杂

科学：arXiv（200万论文，含 LaTeX 源码）；Semantic Scholar ORC

金融：SEC EDGAR（上市公司年报）；注意数据时效性和前瞻性声明

数据混合策略

等权混合：各数据源贡献相同数量的 tokens。缺点：数量少的高质量数据会被淹没。

质量加权混合：给高质量数据更高的采样概率（如维基百科 5×权重，CommonCrawl 1×权重）

温度采样混合：概率 ∝ n_i^α，α<1 时平滑分布，防止大数据集垄断。LLaMA 用 α=0.7

实践建议：先用等权混合验证管线，再用质量加权做最终训练

数据飞轮：用模型生成下一版训练数据

v1 模型 → 在线部署 → 收集真实用户问题 → 人工标注好回答 → v2 SFT 数据 → v2 模型

这是 InstructGPT 和后续所有商业模型的核心闭环。即使是 1000 条真实用户问题，其领域分布和表达方式也比合成数据更真实，往往带来质的提升。

小技巧：在模型输出里加入"对以上回复满意吗？[是/否]"按钮，自动收集偏好信号用于 DPO。

数据质量 > 数据数量：100万条低质量数据（重复、错误、乱码）的效果往往差于 10万条高质量数据。推荐先用小数据集跑通流程，验证 loss 下降后再扩规模。特别注意：SFT 数据里的错误信息会被模型记住并重复，质量问题比量的不足更难修复。

数据质量处理流程

原始数据直接训练效果往往很差。这里是实际项目中常用的数据清洗流程，从算法原理到实用阈值，按优先级排序。

Step 1: MinHash LSH 去重（最重要，优先做） 原理：每条文本 → 提取 n-gram (通常 5-gram) → 每个 n-gram 哈希 用 b 组 r 个哈希函数计算 MinHash 签名（b×r 维向量） 两文本 Jaccard 相似度 ≥ θ 时，LocalitySensitiveHashing 碰撞概率高 为什么不用 pairwise 相似度？ N 篇文档两两比较：O(N²) 复杂度 10M 文档 → 10^14 次比较 → 不可行 MinHash+LSH：O(N) 复杂度，可以处理万亿级数据集 调参建议： band=20, row=5 → 签名长度100，相似度阈值 ≈ 0.7（宽松去重） band=20, row=10 → 签名长度200，相似度阈值 ≈ 0.8（严格去重） 假阳性率（误删不重复文本）与 band/row 的关系：更多 band → 更低假阳性 工具：pip install datasketch text-dedup Step 2: 长度过滤 丢弃 <50 tokens（太短，没有完整信息） 丢弃 >4096 tokens（超长文本往往是爬虫抓到的杂乱内容） 对 SFT：保证回复长度 >20 tokens（太短的回复质量差） 对 SFT：统计回复长度分布，若 P50 < 50 tokens，数据质量存疑 Step 3: 语言检测（fasttext） fasttext 语言检测原理： 字符 n-gram（1-3 gram）作为特征 → 线性分类器 176 种语言，准确率约 99.5%（单语文本） 代码混合文本（英文+中文）准确率下降，建议阈值 >0.8 才保留 工具：import fasttext; lid = fasttext.load_model("lid.176.ftz") Step 4: 内容过滤 关键词黑名单（广告词、侮辱性词汇） 去除纯数字、纯符号行 去除低信息密度内容（重复的导航栏文本、版权声明） NSFW 过滤：Perspective API 或开源 detoxify 库 Step 5: 困惑度（Perplexity）质量过滤 做法：用少量人工标注的"高质量文本"训练一个 KenLM 4-gram 语言模型 对每篇文档计算 PPL；PPL > 阈值的文档质量差 直觉：PPL 高 = 语言模型"觉得"这段文本不自然，可能是乱码/重复/低质 阈值：英文通常 PPL < 300；中文 PPL < 200（具体取决于参考模型） 工具：pip install kenlm
三种去重粒度：
· 文档级去重：整篇文章的 MinHash，粒度粗，最快。适合预训练语料
· 段落级去重：每段独立计算，更细粒度。适合处理部分重复的文档（如新闻转载只改了开头）
· n-gram 级去重（suffix array）：找出所有跨文档共享的 13-gram，最细最慢。Google 用于 C4 数据集的"token-level dedup"
建议：预训练用文档级，SFT 数据用段落级即可。

有害内容检测管线

三层防御：

第1层 关键词黑名单（约10ms/文档）：维护一个 ~5000 词的黑名单，命中即丢弃。召回率低但精确率高

第2层 ML 分类器（约100ms/文档）：Perspective API（Google），或 HuggingFace 上的 toxic-bert。AUC 约 0.95，适合大规模过滤

第3层 人工审核（边缘案例）：对 ML 分类器打分在 0.3-0.7 之间的文档进行人工审核。约占总数据的 2-5%

Tokenizer 覆盖率检查

清洗后，在正式训练前必须做此检查：

unk_rate = count(UNK) / total_tokens

· UNK 率 >5%：tokenizer 覆盖不足，需要扩词表或换 tokenizer
· UNK 率 1-5%：可接受，但建议增加相关语言的预训练语料
· UNK 率 <1%：良好

另外检查：单字符 token 占比（如果很高，说明 BPE merge 不充分，压缩率差，会浪费 context window）。

量化质量目标

预训练语料清洗后的健康指标：

· 平均文档长度：200-2000 tokens（太短则信息密度低）
· 词汇量（unique tokens）：>80% 的词表被覆盖
· 去重后保留率：通常 60-80%（保留率太高说明去重不彻底）
· 语言纯净度：目标语言占比 >95%
· 困惑度中位数：参考良好中文语料，PPL 中位数约 50-150

SFT 数据的健康指标：
· 回复长度中位数：>50 tokens
· 指令多样性（distinct n-gram）：防止模板化回复

实用工具链

text-dedup（pip install text-dedup）：MinHash+LSH，支持 HuggingFace datasets 格式，可直接处理 100M+ 文档

fasttext：语言检测，比 langdetect 快 100×

kenlm：n-gram 语言模型，困惑度过滤，C4 和 CCNet 都用它

datasketch（pip install datasketch）：MinHash 算法的 Python 实现，更灵活

dolma-toolkit：Allen AI 开源的完整数据处理管线，包含所有上述步骤

实用小技巧：与其花大量时间清洗低质量数据，不如直接使用高质量数据源（如知乎高赞回答、维基百科、教科书、技术文档）。这些来源的噪声天然很低。把 80% 的精力用于数据获取，20% 用于清洗，往往比反过来效果更好。

训练配置调参

超参数选择是训练中最需要经验的部分。这里从理论推导到实用规则，覆盖每个关键超参。

线性缩放规则的推导： batch_size 增大 k 倍时，每步梯度估计的方差变为 1/k（中心极限定理）。为保持相同的"噪声水平"，可以把学习率增大 k 倍（线性缩放），或增大 √k 倍（更保守的平方根规则）。实践中，batch 从 32 增大到 256（8×），通常学习率乘 2-4× 而不是 8×，因为过大的 LR 会导致不稳定。Facebook 在 ResNet 论文里验证了这个规则在 batch≤8192 时有效。

学习率 — 理论与实践

预训练：5e-4（大规模 batch 时可以用更大的 lr）
SFT：1e-4 到 5e-4（从预训练 lr 的 1/3 开始）
RLHF/DPO：5e-5 到 1e-4（更小，避免破坏 SFT 学到的对话能力）

LR Finder 技术：
从 lr=1e-7 开始，每步乘以 1.1，记录每步 loss。在 loss 开始反弹前的 lr 即为最大可用 lr，取其 1/10 作为初始 lr。这是 fast.ai 推广的技术，5-10 分钟找到最优 lr。

Adam 优化器内部机制

m_t = β1·m_{t-1} + (1-β1)·g_t ← 一阶矩（动量）
v_t = β2·v_{t-1} + (1-β2)·g_t² ← 二阶矩（自适应LR）
θ = θ - α·m_t/√(v_t+ε)

· β1=0.9（默认），β2=0.95（比0.999更快跟踪梯度变化）
· Weight decay 不应用于 bias 和 LayerNorm 参数（这些参数不需要正则化）
· ε=1e-8 防止除零，bfloat16 下建议用 1e-5 避免精度问题

FP16 vs BF16 选择

FP16（IEEE half-precision）：
· 指数位：5位，范围 ±65504
· 尾数位：10位
· 溢出风险高（Attention logits 可能超过65504）
· 需要 GradScaler 动态缩放防止溢出
· 优点：V100 等老 GPU 原生支持，更广泛的硬件兼容性

BF16（Brain Float 16）：
· 指数位：8位（与FP32相同），范围极大
· 尾数位：7位（精度略低于FP16）
· 无溢出风险，训练更稳定
· A100/H100 首选，RTX 30/40 系列支持

batch size 与显存

越大越好（梯度估计更准，训练更稳定），但受限于显存。

显存不够时用梯度累积模拟大 batch：
batch=8, accum=4 → 等效 batch=32

注意：梯度累积时，每个 micro-batch 单独前向+反向，不用 torch.no_grad()，最后统一 optimizer.step()。累积过程中记得 loss 除以 accum_steps 归一化。

梯度范数监控

"健康"的梯度范数（grad_norm）：

· 训练初期（step 1-100）：1.0-10.0（正常，参数随机初始化）
· 正常训练中：0.1-2.0
· >5.0 持续出现：学习率过大或模型不稳定
· >100 或 inf/nan：梯度爆炸，立即降低 lr

添加监控：
norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
wandb.log({"grad_norm": norm.item()})

序列长度课程学习

渐进式序列长度策略（显著提升训练效率）：

阶段1 seq=128，epoch 1：用 4× 更多样本学习基础语言能力
阶段2 seq=256，epoch 2：学习短段落理解
阶段3 seq=512，epoch 3-5：最终目标长度

为什么有效：短序列时同等显存可以用 4× 的 batch size，梯度估计更准，学习基础语法更快。GPT-3 和 LLaMA-2 都用了这个技巧。

训练轮数与过拟合检测

预训练：1-2 epoch（数据够多时）
SFT：3-5 epoch（数据少时）
RLHF：1-2 epoch

过多 epoch = 过拟合信号：
· train_loss 持续下降但 val_loss 开始上升
· 模型开始"背诵"训练数据（用训练集问题测试输出完全一致）
· 生成多样性下降（同一 prompt 反复给出几乎相同的回答）

对策：Early stopping；Dropout 0.1；数据增强

Batch 组成与课程学习

随机混合（默认）：每 batch 随机采样，简单但忽略难度差异

难度课程学习：先喂简单样本，后期引入困难样本
· 按 seq_len 排序从短到长
· 按困惑度排序从低到高
· 效果：早期学习更高效，最终精度相当或略好

Hard Example Mining：周期性识别模型仍然预测错误的样本，在下一个 epoch 中上采样这些样本（类似 AdaBoost 的思想）。

常见问题诊断

训练不顺时，按照以下清单逐项排查。每个症状对应不同的根本原因，快速定位可以节省数小时的时间。

健康训练曲线的形状特征： 初始 loss 应该接近 ln(vocab_size)（随机猜测的熵上界）。之后经历三个阶段： (1) 快速下降期（前5-10%步数）：模型学习基本统计规律，每步 loss 显著下降； (2) 稳定下降期（中间80%）：loss 平滑下降，偶有小幅震荡，这是正常现象； (3) 收益递减期（后10-15%）：loss 下降变缓，接近模型容量上限。 异常信号：早期 loss 下降后突然上升（lr 过大触发梯度爆炸）；loss 平台很长不动（lr 过小或数据有问题）；周期性尖峰（batch 里有异常样本）。
问题：loss 从一开始就不下降（停在 ln(vocab_size) ≈ 8.8） 可能原因： 1. 学习率太小（lr < 1e-6），梯度几乎没更新 2. 数据 tokenize 错误，label 全为 -100 3. 模型没有正确加载权重（随机初始化） 诊断：打印 optimizer.param_groups[0]['lr'] 和 labels.sum() 问题：loss 急剧波动（NaN 或 inf） 可能原因： 1. 学习率太大，梯度爆炸 2. 数据里有特殊字符导致 tokenize 异常（产生超长序列） 3. FP16 下 Attention logit 溢出（改用 BF16 或降 lr） 解决：降 lr 10×，检查数据，确认 clip_grad_norm max_norm=1.0 问题：模型会说话但回答永远跑题/重复 可能原因： 1. SFT 数据的 label mask 没有正确设置（prompt 部分没有 mask 掉，模型在学对问题） 2. SFT 数据质量差（回答都是套话/模板） 3. repetition_penalty 未设置（推理时加 rp=1.1-1.3） 诊断：打印一个 batch 的 labels，检查 -100 的分布比例（应该约 60-70%） 问题：模型只输出英文，即使训练数据是中文 可能原因： 1. Tokenizer 词表覆盖中文字符不足（UNK 率高） 2. Embedding 层权重没有正确初始化中文词汇 3. 预训练权重来自英文模型，中文字符的 embedding 是随机的 修复：检查 unk_rate；若 >10%，需要从头训练或更换 tokenizer 问题：模型回答总是很短 可能原因： SFT 数据集中答案长度分布偏短（如大量只有一行的答案） 模型学到"短答案是高频的"，倾向于提早生成 EOS 修复：统计 SFT 数据回复长度分布；剔除或降采样 <50 token 的回复 补充一些长格式回复（详细解释、分步骤回答） 问题：loss 很低但模型输出是乱码/重复 可能原因： 标签泄漏（Label Leakage）：模型在输入里已经看到了答案 训练时 input_ids 和 labels 对齐错误（off-by-one shift 问题） 诊断：打印 input_ids[0] 和 labels[0]，确认 labels 是 input_ids 的右移一位 问题：训练初期正常，后期质量突然下降 可能原因： 学习率在 warmup 后太高，后期出现梯度震荡 余弦衰减的最低点（min_lr）设置过高 修复：检查 lr_scheduler 曲线；将 min_lr 设为 max_lr 的 10% 问题：OOM（CUDA Out of Memory） 快速解决（按效果排序）： 1. 减半 batch_size，增倍 accumulation_steps 2. 开 gradient_checkpointing=True（内存减半，速度降约 30%） 3. 减小 max_seq_len（512 → 256） 4. 用 bfloat16 代替 float32 5. 使用 DeepSpeed ZeRO-2 分片优化器状态

梯度流调试技巧

用 register_hook 可视化每层梯度大小：

grad_norms = {}
for name, p in model.named_parameters():
  if p.requires_grad:
    p.register_hook(lambda g, n=name:
      grad_norms.update({n: g.norm().item()}))

健康模式：各层梯度范数相近（0.01-1.0），浅层略大。若某层梯度为0，可能是该层被冻结或存在梯度消失。

激活值统计监控

用 forward hook 检测异常激活：

· Dead neurons（激活值恒为0）：GELU/SwiGLU 下较少见，但若 >20% 的神经元 activation 均值接近0，说明该层学习不充分

· 激活值饱和：若 LayerNorm 输入的均值 >10 或方差 >100，说明数值不稳定，检查 embedding 初始化

· 在每100步打印一次各层的 activation mean/std，可以早期发现数值问题

swanlab / wandb 应追踪的指标

必须追踪：
· train_loss, val_loss（每100步）
· learning_rate（实际值，非初始值）
· grad_norm（每步）

推荐追踪：
· token_accuracy（非 loss，直觉更好）
· sequence_length_mean（验证课程学习）
· gpu_memory_allocated
· tokens_per_second（监控训练效率）

可选：
· 每1000步生成一个固定测试 prompt 的输出（直观感知质量变化）

Checkpoint A/B 测试方法

自动化评估（lm-eval）：
对两个 checkpoint 跑相同基准（HellaSwag/MMLU），比较分数差

人工盲评：
准备 20 个固定测试问题，对两个模型的输出随机打乱顺序，由评测人员在不知道来源的情况下选择更好的回答。胜率 >60% 视为有显著改进。

自动指标：
ROUGE-L（摘要任务）、BLEU（翻译）、Exact Match（问答）

优化 1：Flash Attention 2

现状：代码用 F.scaled_dot_product_attention（Flash Attn 1 的 PyTorch 版）
改进：安装 flash-attn 包，改用 flash_attn.flash_attn_func
效果：seq=2048 时速度提升 2-4×，显存节省 50%（不存储整个 attention matrix）
原理：分块计算 attention，每次只将一个块加载到 SRAM（快存储），避免反复读写 HBM（慢存储）
成本：pip install flash-attn（约10分钟编译），改 5 行代码

优化 2：torch.compile

现状：代码支持 --use_compile 1 但默认关闭
改进：开启 torch.compile(model, mode="reduce-overhead")
效果：训练速度提升 20-40%（消除 Python 解释器开销，生成优化的 CUDA kernel）
模式选择："default" 最通用；"reduce-overhead" 适合固定 shape；"max-autotune" 编译时间长但最快
成本：第一个 batch 需要编译（约 1-2 分钟），后续全部加速

优化 3：梯度检查点（Gradient Checkpointing）

原理：正常训练时，所有层的激活值都保存用于反向传播。梯度检查点不保存中间激活，反向传播时重新计算。
效果：激活内存减少约 √num_layers 倍（8层减少约 2.8×），代价是额外约 33% 的前向计算
使用：model.gradient_checkpointing_enable()（HuggingFace API）或在 nn.Module 里用 torch.utils.checkpoint.checkpoint()
适用场景：显存不足时的首选解决方案，几乎没有代码改动成本

优化 4：增加词表大小

现状：vocab_size=6400（极小，字符级 + 少量词组）
改进：用 sentencepiece/tiktoken 训练 32000 或 64000 词表
效果：中文 token 数减少 30-50%（"北京" 从 2 tokens 变 1 token），可以处理更长的上下文，支持多语言
代价：embedding 层参数量增加（32K×768 = 24.6M vs 6.4K×768 = 4.9M），需要重新训练 tokenizer

优化 5：Sequence Packing

问题：每条样本 pad 到 max_length，大量无效 padding token 参与计算
解决：把多条短文本拼接成一条长序列，中间用特殊分隔符，用 block-diagonal attention mask 隔开不同样本
效果：有效 token 比例从 30-60% 提升到 90%+，等于以相同计算量处理了 2× 的实际数据
成本：需要修改 dataset 和 collate_fn，并且 attention 需要支持 cu_seqlens 参数（Flash Attention 2 支持）

优化 6：MoE（混合专家）

现状：代码支持 --use_moe 1，4个专家 Top-1 路由
效果：参数量翻 4 倍但每步计算量不变，等于用 4× 的模型容量处理信息
挑战：负载均衡（防止所有 token 都路由到同一个专家）；Expert Collapse（部分专家从不被使用）
负载均衡损失：L_bal = α · Σ(f_i · p_i)，f_i 是专家使用频率，p_i 是路由概率

优化 7：知识蒸馏（Knowledge Distillation）

目标：训练小模型模拟大模型的行为，实现 90% 的能力用 30% 的参数
方法：蒸馏损失 = α×CE(hard labels) + (1-α)×KL(soft labels)，soft labels 是大模型的 token 概率分布（温度 T=2-4 软化）
效果：DistilBERT（BERT 40% 大小，97% 性能）；DistilGPT-2（GPT-2 Small 70% 大小）
前提：需要有一个训练好的 teacher 模型（可以是 GPT-4 的 logprobs 或开源大模型）

优化 8：LoRA 持续学习

问题：给已训练模型加新领域时，全量微调会"灾难性遗忘"旧能力
LoRA 方法：冻结原始权重 W，只训练低秩分解 ΔW = A·B（rank r≪d），参数量约为全量的 0.1-1%
优点：可以同时维护多个 LoRA adapter（每个领域一个），推理时按需加载，不遗忘
工具：pip install peft（HuggingFace PEFT 库，5 行代码集成 LoRA）

效率指标

Tokens per Dollar（TPD）：衡量训练性价比
RTX 4090（Vast.ai）：60K tok/s ÷ $0.3/h ÷ 3600 = 200K tok/$，最高
A100（Lambda）：180K tok/s ÷ $2.5/h ÷ 3600 = 72K tok/$
H100（Lambda）：320K tok/s ÷ $3.5/h ÷ 3600 = 91K tok/$

MFU（Model FLOPS Utilization）：实际 FLOPS / 峰值 FLOPS
优化后的训练约 40-60% MFU。低于 30% 说明有优化空间（通常是 I/O 瓶颈）

何时扩大规模？决策框架

验证数据后再扩规模：先用 1% 的数据跑一个小模型，确认 loss 正常下降，再投入全量资源

观察 val_loss 平台：若 val_loss 下降变缓（连续 5000 步下降 <1%），是扩大数据量的信号

缩放律预测：根据 Chinchilla 规律，预测全量数据下的最终 loss，若与小模型实验的趋势一致，可以安全扩规模

不要过早扩规模：数据问题在小实验里会同样出现，先解决再扩
